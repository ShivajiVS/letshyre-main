1. Candidate Flow Overview

The candidate flow represents the complete user journey from first landing on the platform to actively using the job matching dashboard. This document outlines every screen in the flow and specifies exactly what data must be provided by the backend to make each screen dynamic and functional.

📌 Important Note: This flow assumes the backend is fully built and provides REST APIs that deliver the specified data in JSON format. All screens should be designed to handle data loading states, error states, and empty states gracefully.

Candidate Flow Stages:

  1. Landing Page - Platform introduction & feature showcase
  2. Authentication Entry - Choose between login, register, or employer
  3. Login - Credential verification
  4. Forgot Password - Password recovery workflow
  5. Registration - Multi-step account creation
  6. Profile Completion - Setup profile after login
  7. Dashboard - Main job browsing interface

2. Screen-by-Screen Data Requirements

🔑 Key Convention: REQUIRED (Red) fields must be present in every response. OPTIONAL fields may be conditionally included.

3. Landing Page Data

🏠 Screen: Landing.jsx

Purpose: Introduce the platform, showcase features and benefits, encourage sign up

Data Requirements:

The landing page is primarily static content with some dynamic hero sections and testimonials. The required data from backend:

Data Element Type Status Description API Endpoint
Hero Banner Images Array[String] OPTIONAL URLs of rotating banner images (3-5 slides) GET /cms/v1/hero-banners/
Feature Cards Array[Object] OPTIONAL Platform features with icons, titles, descriptions GET /cms/v1/features/
Testimonials Array[Object] OPTIONAL User success stories with names, photos, quotes GET /cms/v1/testimonials/
FAQ Items Array[Object] OPTIONAL Frequently asked questions and answers GET /cms/v1/faqs/
Video URL String OPTIONAL Embedded video link (YouTube/Vimeo) GET /cms/v1/video-content/

Response Structure Example:

GET /cms/v1/hero-banners/ { "status": "success", "data": [ { "id": 1, "image_url": "https://cdn.letshyre.com/banner1.jpg", "title": "Find Your Dream Job Today", "description": "Explore thousands of opportunities", "cta_button": "Get Started", "cta_link": "/get-started" }, { "id": 2, "image_url": "https://cdn.letshyre.com/banner2.jpg", "title": "Build Your Career", "description": "Connect with top employers", "cta_button": "Sign Up", "cta_link": "/register/candidate" } ] }

Screen Behavior:

✓ Loading State: Show skeleton loaders for hero, feature cards, and testimonials
✓ Error State: Display generic static content if API fails (fallback UI)
✓ Success State: Render fetched data with auto-rotating carousel for hero

Purpose: Authenticate candidate with email and password

Data Element Type Purpose

Login Response (Backend → Frontend):

Error Response:

POST /user/v1/login/ - Error (401) { "status": "error", "error": { "code": "INVALID_CREDENTIALS", "message": "Invalid email or password", "detail": "Email or password is incorrect. Please try again." } }

Frontend Data Handling After Login:

Store in localStorage:
  • accessToken - JWT token for API calls
  • refreshToken - Token for refreshing access token
  • user - User object as JSON string
Check Profile Status:
  • If profile_completed === false → Show ProfileCompletion flow
  • If profile_completed === true → Redirect to dashboard directly

Error States to Handle:

⚠️ Possible Error Codes:
  • INVALID_CREDENTIALS - Wrong email/password
  • ACCOUNT_INACTIVE - Account disabled
  • ACCOUNT_NOT_VERIFIED - Email not verified
  • TOO_MANY_ATTEMPTS - Account temporarily locked
  • SERVER_ERROR - Network/server issue

Dashboard State Management Data:

State/Data Type Purpose
profileCompleted Boolean Determine if profile is 100% complete → unlock restricted tabs
activeTab String Track which tab user viewing (jobs, applications, scorecard)
showProfileFlow Boolean Show profile completion overlay if profile not complete
userInfo Object Current user details for display in topbar

Dashboard Sidebar Sections:

Section 1: Find Jobs
  • Always unlocked
  • No profile required
Section 2: My Applications
  • Locked if profileCompleted === false
  • Shows: Applications count, status indicators
Section 3: Scorecard
  • Locked if profileCompleted === false
  • Shows: Overall score, ratings from employers
Section 4: Logout
  • Always available
  • Clear tokens & redirect to login

14. Data Consistency Guidelines

🔄 Frontend-Backend Data Synchronization

Error Handling Standards:

⚠️ All Error Responses Should Follow This Format:
{ "status": "error", "error": { "code": "ERROR_CODE", "message": "User-friendly error message", "detail": "Technical details if applicable", "field": "field_name" (if field-specific) } }

Common Error Codes:

Error Code HTTP Status Meaning Frontend Action
INVALID_CREDENTIALS 401 Email or password incorrect Show error message, keep form
EMAIL_ALREADY_EXISTS 409 Email already registered Suggest login or forgot password
INVALID_OTP 400 OTP code incorrect Show error, allow retry
OTP_EXPIRED 400 OTP code expired Offer to resend OTP
UNAUTHORIZED 401 No valid token Redirect to login
PROFILE_NOT_COMPLETE 403 Profile completion required Show profile stepper overlay
SERVER_ERROR 500 Backend server error Show generic error, retry button

Loading State Guidelines:

✓ Show skeleton/spinner during:
  • Fetching job listings
  • Searching/filtering jobs
  • Loading applications list
  • Loading scorecard data
  • API calls on profile completion steps

Empty State Guidelines:

✓ Show empty state message when:
  • No jobs found after search/filter
  • No applications submitted yet
  • No ratings/scorecard available
  • Search returns zero results
Message format: "No {items} found. {Action or suggestion to find items}"

Token Expiration & Refresh Flow:

1. Frontend makes API request with access token 2. Backend validates token ├─> If valid: Process request normally └─> If expired (401): Return 401 with error 3. Frontend intercepts 401 response 4. Frontend calls POST /user/v1/login_refresh/ with refresh token 5. Backend validates refresh token ├─> If valid: Return new access token │ └─> Frontend retries original request with new token └─> If expired/invalid: Return 401 6. Frontend handles refresh failure ├─> Clear tokens from localStorage ├─> Redirect to /login/candidate └─> Show "Session expired" message

Data Caching & Update Patterns:

Cache Strategy:
  • User profile: Cache for 5 minutes or until logout
  • Jobs list: Cache for 2 minutes, refresh on tab focus
  • Applications: Cache for 2 minutes, refresh on status change
  • Scorecard: Cache for 5 minutes, no real-time updates needed
Invalidation Triggers:
  • User completes profile → Invalidate dashboard state
  • User applies to job → Invalidate jobs list + applications list
  • User updates profile → Invalidate profile cache

Real-Time Updates (WebSocket Considerations):

For Future Implementation:
  • Application status changes (pending → shortlisted)
  • New ratings received (employer feedback)
  • Messages from employers (if messaging feature added)
  • Job matches (if recommendation engine added)

Data Validation Rules (Frontend):

Before Submitting to Backend:
  • Email: Valid format (RFC 5322)
  • Password: Min 8 chars, at least 1 uppercase, 1 lowercase, 1 number, 1 special char
  • Phone: Valid international format
  • Age: Calculated from DOB must be >= 18
  • Resume: File size <= 5MB, format: PDF/DOCX/DOC only
  • Salary: Min <= Max, both > 0

Pagination Standards:

Default Pagination Values:
  • Page: 1 (first page)
  • Limit: 10 items per page (configurable)
  • Response includes: total_count, page, limit, total_pages
  • Fetch next page: Increment page parameter

Sorting & Filtering Standards:

Query Parameter Format:
  • Sort Ascending: sort=field_name
  • Sort Descending: sort=-field_name
  • Filter: filter_name=value
  • Multiple Filters: ?status=pending&category=tech&sort=-date
